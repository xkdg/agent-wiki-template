# CLAUDE.md

Стартовый промпт агента. Не содержит проектных фактов — конкретику смотри в `wiki/INDEX.md`.

---

## Протокол агента

**Перед каждой задачей читать:**
1. `wiki/INDEX.md` — точка входа в проект: что за проект, карта сервисов, ключевые ограничения, роутинг по задачам
2. `wiki/context/QUICK_REF.md` — cheat sheet с оперативными фактами (state machine, формулы, пороги)

**По запросу или при необходимости** (не на каждой задаче):
- `wiki/architecture/OVERVIEW.md` — архитектура, C4, ER, cross-cutting concerns
- `wiki/adr/` — архитектурные решения с обоснованием
- `wiki/context/PATTERNS.md` / `ANTI_PATTERNS.md` — правила «как делать» / «как не делать»

**После задачи** — две независимые проверки:
1. Прогон по чек-листу «архитектурно значимая задача» (см. ниже). Если хоть один пункт `да` → пройти по матрице документации, **независимо от объёма правки**.
2. Если задача велась по ТЗ-файлу — отметить `[x]` выполненные пункты, обновить статус.

## Подход к работе

**1. Думать до кода.** Не прятать неопределённость. Если несколько интерпретаций — озвучить, не выбирать молча. Если проще → сказать и обосновать. Если непонятно → остановиться и спросить.

**2. Минимум кода.** Только то, что закрывает задачу. Никаких «гибких» абстракций под единственный use-case, никакого error-handling под невозможные сценарии. Если написал 200 строк, а можно 50 — переписать.

**3. Хирургические правки.** Трогать только то, что нужно по задаче. Не «улучшать» соседний код, не рефакторить то, что не сломано, не удалять предсуществующий мёртвый код без явной просьбы. Держаться существующего стиля файла. Каждая изменённая строка должна трассироваться к запросу.

**4. Цели с проверяемым критерием.** «Исправить баг» → «написать тест, который его воспроизводит, сделать зелёным». «Добавить валидацию» → «тесты на невалидный вход + зелёные». Для многошаговых — краткий план с чек-поинтами.

**5. Не замалчивать.** При блокере — написать, что блокирует и какие варианты есть. При найденных по пути проблемах, не входящих в задачу, — упомянуть в финале, не править молча.

## Архитектурно значимая задача — чек-лист

**Объём кода не критерий значимости.** Однострочная правка может быть значимой; большая задача — нет. Прогон по чек-листу делается после **любой** задачи. Если хотя бы один пункт `да` — обновление документации обязательно по матрице.

- [ ] Меняется **state machine**: финальные статусы, флаги, переходы между ними
- [ ] Меняется **схема БД**: DDL, миграции, новые/удалённые/переименованные колонки или индексы
- [ ] Меняется **контракт API или CLI**: новые/удалённые/изменённые флаги, параметры, формат вывода
- [ ] Меняется **протокол между сервисами или публичный API `shared/`**: новые/удалённые/переименованные функции, типы, error codes
- [ ] Меняются **конфиги, пороги или формулы**, влияющие на поведение пайплайна
- [ ] Меняется **инвариант, описанный в ADR** (даже если правка одной строкой кода)
- [ ] Меняется **cross-cutting функциональность**: обработка ошибок, retry, логирование, healthcheck

**Конкретизация под текущий проект** (какие именно файлы и сущности подпадают под каждый пункт) — в `wiki/INDEX.md §Архитектурно значимое в этом проекте`.

## Документация — что писать после задачи

| Тип работы | Обязательная документация |
|---|---|
| Bug fix с нетривиальной диагностикой (>30 мин расследования или неочевидный симптом) | Post-mortem в `wiki/decisions/YYYY-MM-DD-<сервис>-<тема>.md`: симптом → диагностика → root cause → фикс |
| Найден новый антипаттерн (правило «не делайте так» с прецедентом) | + запись в `wiki/context/ANTI_PATTERNS.md` |
| Использован/изобретён переиспользуемый приём (повторил то, что точно сделаю снова) | Запись в `wiki/context/PATTERNS.md` |
| Архитектурное решение (выбор из явных вариантов, влияет на >1 сервис или на схему данных) | ADR в `wiki/adr/`: контекст → варианты → решение → последствия |
| Изменена state machine / формулы / пороги | Обновить `wiki/context/QUICK_REF.md` |
| Изменена схема БД / структура модулей / cross-cutting | Обновить `wiki/architecture/OVERVIEW.md` |
| Сознательный «костыль на потом» | Запись в `wiki/todo/TECH_DEBT.md` |
| Завершена задача из любого ТЗ-документа (BACKLOG, DEV_SPEC, PLAN и т.п.) | Отметить `[x]` **прямо в исходном документе**, не плодить отдельный архив |
| Новая задача / идея, не делаем сейчас | Запись в `wiki/todo/BACKLOG.md` |

**Принцип амортизации.** `decisions/` — сырьё. По мере того, как урок переезжает в долгоживущий артефакт (ADR / PATTERNS / ANTI_PATTERNS), исходный decision можно удалять. Дублировать смысла нет.

**Принцип единственного источника истины.** Каждый факт живёт в одном месте + cross-refs в смежных. Дубль текста = долг, при правке одной копии остальные молча устаревают.

**Принцип cross-references.** У большинства паттернов есть зеркальный антипаттерн и наоборот. В каждом блоке — строка `См. также:` со ссылками на связанные записи.

**Принцип «отметка на месте, не архив».** Любой ТЗ-документ после выполнения задачи получает `[x]` **прямо в исходном файле**. Не создавать отдельные `COMPLETED_*.md` архивы. Если у задачи появилась переиспользуемая деталь реализации — она переезжает в долгоживущий артефакт (ADR / PATTERNS / ANTI_PATTERNS), и в исходном ТЗ остаётся одна строка `[x]` с короткой ссылкой. Запись можно удалить позже, если архивная ценность нулевая.

## ТЗ — границы и формат

ТЗ-файл и обновление документации — **две независимые оси**. ТЗ-файл нужен для планирования и трекинга крупной/неочевидной работы. Обновление документации — обязательная пост-проверка любой архитектурно значимой задачи (см. чек-лист выше). Маленькая правка может быть значимой и потребовать обновления вики без создания ТЗ-файла; большая задача может не трогать архитектуру, но требовать ТЗ-файла для декомпозиции.

**Когда оформлять как ТЗ-файл** в `wiki/todo/<short-kebab-name>.md`:
- Задача больше пары часов работы или требует декомпозиции на шаги
- Скоуп нечёткий, нужно явно зафиксировать «что в, что вне» до старта
- Нужно согласование подхода с пользователем до реализации

**Границы одного ТЗ:**
- **Один связный кусок работы.** Один сервис или одна подсистема `shared/`. Если задача затрагивает >1 сервиса — это либо серия ТЗ с явными зависимостями, либо ADR-уровневое решение (тогда сначала ADR).
- **Soft-лимит ~150-200 строк файла.** Больше — сигнал к декомпозиции.
- **Никакого псевдокода и реализаций функций.** ТЗ описывает «что» и «почему», не «как». Исключение — нетривиальные edge cases или контракт API.

**Шаблон:** `wiki/todo/_TEMPLATE.md`. Скопировать, переименовать, заполнить.

**Обязательные секции** (в этом порядке): Контекст · Скоуп (что входит / что НЕ входит) · Перед стартом — изучить · План реализации · Критерии приёмки · После реализации обновить.

**Опциональные секции** (удалять, если пустые): Открытые вопросы · Риски.

**Жизненный цикл:** `draft` → `in-progress` → `done`. После `done` файл может быть удалён, если уникальной ценности не осталось (всё уже в коде + git log + долгоживущих артефактах). Файл сохраняется только если несёт самостоятельный исторический контекст.

## Структура wiki/ (шаблон ролей)

```
wiki/
├── INDEX.md                       ← точка входа в проект (всегда читается перед задачей)
├── context/
│   ├── QUICK_REF.md              ← cheat sheet: state machine, формулы, пороги (всегда читается)
│   ├── PATTERNS.md               ← прескриптивно «как делать»
│   └── ANTI_PATTERNS.md          ← дескриптивно «как не делать», с first cause
├── architecture/
│   └── OVERVIEW.md               ← Explanation (Diátaxis): C4, pipeline, ER, tech stack, cross-cutting
├── adr/                          ← архитектурные решения, нумерованные, постоянные
│   ├── _TEMPLATE.md
│   └── 00NN-<тема>.md
├── decisions/                    ← post-mortem по инцидентам, амортизируется
│   ├── _TEMPLATE.md
│   └── YYYY-MM-DD-<сервис>-<тема>.md
└── todo/
    ├── _TEMPLATE.md              ← шаблон ТЗ (копировать → заполнить)
    ├── BACKLOG.md                ← планируемое (выполненные отмечаются [x] на месте)
    ├── TECH_DEBT.md              ← осознанные долги (статусы: accepted/planned/partial)
    └── <task-name>.md            ← конкретные ТЗ по задачам
```

**Жанровая граница:**
- **OVERVIEW** — *объяснение* «почему именно так», читается периодически. Длинно, диаграммы, обоснования.
- **QUICK_REF** — *оперативный cheat-sheet*, читается на каждой задаче. Только то, что плохо запоминается и часто нужно (state machine, ключевые числа).
- **ADR** — *прескриптивное проектное решение*, принятое до/вместе с реализацией. Долгоживущее, заменяется только новым ADR (`supersedes`).
- **decisions** — *дескриптивный лог инцидента*. Привязан ко времени (дата в имени). Сырьё для долгоживущих артефактов.
- **PATTERNS / ANTI_PATTERNS** — *выводы*, правила. Прескриптивная и дескриптивная стороны одного и того же.

## Правила именования и нумерации артефактов

**ADR:** `wiki/adr/00NN-<тема>.md`
- NNNN — sequential, **никогда не переиспользуется** (даже если ADR удалён или superseded — номер не возвращается)
- Замена решения — через новый ADR с пометкой `Supersedes ADR-XXXX` в шапке
- Удалять ADR можно только если решение никогда не было применено и контекст полностью неактуален

**Антипаттерны / паттерны / технический долг:** `A-NN` / `P-NN` / `TD-NN`
- Sequential, **никогда не переиспользуется**. Если запись удалена — номер не возвращается (пропуск в нумерации допустим)
- В заголовках всегда пишется полный ID: `### A-23: <название>`
- Cross-refs указываются через ID: «См. также: P-12, A-22»

**Decisions (post-mortem):** `wiki/decisions/YYYY-MM-DD-<сервис>-<краткая-тема>.md`
- Дата — день фиксации post-mortem (не день инцидента, если они расходятся)
- `<сервис>` — `shared` для общей библиотеки, иначе имя сервиса
- `<краткая-тема>` — kebab-case, 2-4 слова

**BACKLOG / TECH_DEBT — внутренняя нумерация:**
- BACKLOG — задачи списком `[ ] / [x]` под тематическими секциями, без формального ID
- TECH_DEBT — `TD-NN` по тем же правилам, что A/P (sequential, never reused)

**Заголовки записей** — содержат суть в одной строке, не оставляют для тела документа объяснение «о чём это». Идентификатор в заголовке обязателен.
