## Проект

*2–4 предложения: что делает система, для кого, на каком масштабе. Без маркетинга — фактическое описание.*

*Пример:*

*Автоматическое обогащение реестра юридических лиц технологическим стеком и контактами. На входе — список организаций. На выходе — Parquet с верифицированным сайтом, tech stack, контактами и LLM-суммаризацией.*

*Пайплайн: `search` → `aiohttp-crawl` → `playwright-crawl` → `deep-crawl` → `llm-analyze` → `export`. Координация — через PostgreSQL (state machine), без прямых вызовов между сервисами.*

## Ключевые ограничения проекта (не нарушать без ADR)

*Список инвариантов проекта. Это **прескриптивные правила**, нарушение которых требует осознанного ADR. Не «как делать», а «что фундаментально нельзя».*

*Пример:*

1. *Сервисы **не импортируют** друг друга — только общую библиотеку*
2. *Параметры сервиса (`BATCH_SIZE`, тайм-ауты, пороги) — в `services/<name>/config.py`, **не** в `shared/`*
3. *Миграции БД — только через централизованный механизм, не из кода сервисов*
4. *Финальные статусы state machine — фиксированный набор, расширение требует ADR*

## После задачи — что обновить (применённая матрица)

*Краткая версия матрицы из `CLAUDE.md §Документация`. Здесь — с привязкой к именам файлов в этом проекте.*

| Что я сделал | Куда пишу |
|---|---|
| Bug fix с нетривиальной диагностикой | `wiki/decisions/YYYY-MM-DD-<сервис>-<тема>.md` |
| Найден новый антипаттерн с прецедентом | + `wiki/context/ANTI_PATTERNS.md` (новая запись `A-NN`) |
| Использован/изобретён переиспользуемый приём | + `wiki/context/PATTERNS.md` (новая запись `P-NN`) |
| Архитектурное решение (выбор из вариантов, влияет на >1 компонент или схему) | `wiki/adr/00NN-<тема>.md` |
| Изменена state machine / формула / пороги | `wiki/context/QUICK_REF.md` |
| Изменена схема БД / структура модулей / cross-cutting | `wiki/architecture/OVERVIEW.md` |
| Сознательный «костыль на потом» | `wiki/todo/TECH_DEBT.md` (новая запись `TD-NN`) |
| Завершена задача из любого ТЗ-документа (BACKLOG, DEV_SPEC, PLAN) | отметить `[x]` **прямо в исходном файле**, не плодить отдельный архив |
| Новая задача / идея, не делаем сейчас | `wiki/todo/BACKLOG.md` |

**Цепочка амортизации decisions.** Свежий post-mortem живёт в `wiki/decisions/`. Когда урок попал в `ANTI_PATTERNS` / `PATTERNS` / новый ADR — исходный decision удаляется (он стал дублем). См. `CLAUDE.md §Документация` для полного описания принципа.

**Правила нейминга и нумерации** — в `CLAUDE.md §Правила именования и нумерации артефактов`.

## Точки входа в wiki/

```
wiki/
├── INDEX.md                       ← вы здесь
├── context/
│   ├── QUICK_REF.md              ← cheat sheet (читается на каждой задаче)
│   ├── PATTERNS.md               ← P-NN, как делать правильно
│   └── ANTI_PATTERNS.md          ← A-NN, как НЕ делать
├── architecture/
│   └── OVERVIEW.md               ← C4, pipeline, ER, tech stack, cross-cutting
├── adr/
│   ├── _TEMPLATE.md              ← шаблон ADR (копировать → нумеровать)
│   └── 00NN-<тема>.md
├── decisions/                    ← post-mortem (пополняется по инцидентам)
│   └── _TEMPLATE.md
└── todo/
    ├── _TEMPLATE.md              ← шаблон ТЗ
    ├── BACKLOG.md                ← планируемое + [x] отметки выполненного
    ├── TECH_DEBT.md              ← TD-NN, осознанные долги
    └── <task-name>.md            ← конкретные ТЗ по задачам
```
