## Pipeline — контекстная диаграмма

*Если проект — это pipeline. Иначе — заменить на «модули и их вход/выход».*

```
Stage 0: <вход>          → <таблица/файл>
Stage 1: <что делает>    → <вход> → <выход>
Stage 2: …
```

Каждый компонент: читает по статусу/флагу → обрабатывает → пишет → следующий подхватывает.
Детальный поток с input/output — `wiki/architecture/OVERVIEW.md §Pipeline`.

## State machine

*Все финальные статусы и флаги, которые описывают переход. Это самое важное для агента — у нас один источник правды, и он здесь.*

### `<table>.<status_field>` (N финальных значений)

| Статус | Смысл |
|---|---|
| *`PENDING`* | *необработанный кандидат* |
| *`PROCESSED_OK`* | *успешно прошёл стадию X* |
| *`REJECTED`* | *отбракован по условию Y* |
| *`SKIPPED`* | *обойдён по условию Z* |

### Флаги `<table>` (булевы поля переходов)

| Флаг | Смысл |
|---|---|
| *`stage1_done`* | *Stage 1 завершён* |
| *`needs_stage2`* | *детектирован сигнал для Stage 2* |
| *…* | |

## Ключевые пороги / формулы

*Только числовые константы, которые влияют на бизнес-логику. Где именно в коде они живут — указать. Без обоснования (обоснование — в ADR).*

| Параметр | Значение | Где в коде | ADR |
|---|---|---|---|
| *`WINNER_THRESHOLD`* | *45* | *`services/aiohttp-crawl/src/.../config.py`* | *`ADR-0003`* |
| *`MAX_RETRY_ATTEMPTS`* | *5* | *`services/.../config.py`* | *`ADR-0007`* |

## Гарантии и условия SELECT-ов

*Условия выборки, которые часто требуют изменений и легко сломать. «Все сервисы читают по таким-то условиям».*

```sql
-- aiohttp-crawl выбирает: WHERE verification_status='PENDING'
-- deep-crawl выбирает: WHERE is_winner=TRUE AND deep_crawl_done=FALSE
-- export выбирает: WHERE verification_status='VERIFIED_SCORE'
```
